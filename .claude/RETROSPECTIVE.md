## 根因分析

### 我出错的模式

| 模式 | 表现 | 占比 |
|------|------|------|
| **构建后不测试** | 编译通过就告诉你"可以了"，从不端到端跑 | 70% |
| **没听懂需求就动手** | 你说跨平台我给 Windows API，你说 MSVC 我给 MinGW | 15% |
| **失败→换方案→再失败** | 托盘 6 轮，每轮浅尝辄止不 debug 根因 | 10% |
| **输出噪音远超有效信息** | 每次几百行 Markdown 文档，关键：代码没测 | 5% |

### 为什么"自测"对你没用

我的"自测"环境是 MSYS2 bash + Go + curl，你的运行环境是 **PowerShell + 双击 exe + Windows 托盘区**。我在 bash 里 `go build` 成功不等于你在 PS 里 `.\build.ps1` 能过。我的"测试"从未覆盖你实际执行路径。

## 已立即生效的改进

### CLAUDE.md（项目级规则）

已写入项目根目录 `CLAUDE.md`，包含：
- 环境信息（MSVC路径、Qt路径、CMake路径）
- 编译必须端到端验证
- 先debug后换方案
- 禁止杀系统进程
- 自测流程

### 后续建议（需要你确认）

1. **全局 CLAUDE.md** — `~/.claude/CLAUDE.md`，对所有项目生效
2. **缩减输出** — 每轮回复只给代码/命令/结果，不给 Markdown 表格和报告
3. **执行前清单** — 编译类任务必须先确认环境、先构建、先清理副作用
4. **失败即停** — 一个方案连续失败2次，停止并深度排查，不允许试第3个方案
